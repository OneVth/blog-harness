# RULES

**린터의 계약서다.**

- **여기 있는 것만 코드가 된다.**
- **여기 없는 것은 검사하지 않는다.**

린터 에러는 규칙 ID를 출력한다. 규칙 ID로 이 문서를 찾는다.

---

## 수준

| 수준 | 의미 | 발행 |
|---|---|---|
| **ERROR** | 명백한 위반. 고쳐야 한다 | **차단** |
| **WARN** | 판정이 필요하다. 의도된 것일 수 있다 | 허용 |
| **INFO** | 기록만 한다. 경고가 아니다 | 허용 |

### 판정 원칙

**플래그는 실패가 아니라 질문이다.**

WARN은 세 갈래로 처리한다.

| 결정 | 언제 |
|---|---|
| **fix** | 실수다. 고친다 |
| **confirm** | 의도된 것이다. 그대로 둔다 |
| **promote** | 반복된다. **가이드에 등록한다** |

**promote가 좌상단↔우상단의 피드백 루프다.** WARN이 반복되면 그건 규칙이
잘못된 것이지 사람이 잘못한 게 아니다.

### false positive 금지

**멀쩡한 것을 잡으면 사람이 린터를 무시하게 되고, 그 순간 하네스는 죽는다.**

애매하면 ERROR가 아니라 WARN. WARN도 애매하면 INFO.

## SVG — `lint_svg`

출처: `diagram-system.md`, `diagram-patterns/*.md`

### SVG-01 — viewBox 폭 (ERROR)

```
조건:  viewBox 폭 ≤ 900
```

본문 폭이 900이다 (hELLO `너비: 900`). 초과하면 브라우저가 `max-width: 900px`로
**축소 렌더한다.** 가로 스크롤이 생기는 게 아니라 **이미지가 작아져 텍스트가 안 보인다.**

**폭은 콘텐츠에 맞춘다.** 590~900. 과한 우측 여백을 두지 않는다.

### SVG-02 — 폰트 (ERROR)

```
조건:  루트 <svg> 의 font-family == "Noto Sans CJK KR, sans-serif"
```

### SVG-03 — 금지 유니코드 (ERROR)

```
조건:  ₀₁₂₃₄₅₆₇₈₉  ⁰¹²³⁴⁵⁶⁷⁸⁹  ⋮  없음
```

rsvg-convert + Noto Sans CJK KR에서 **tofu(□)로 깨진다.**

에러 메시지에 대체 문자를 제시한다: `₁` → `1`

> `①`~`⑨`는 정상 렌더된다 (검증됨).

### SVG-04 — XML 주석 (ERROR)

```
조건:  <!-- --> 안에 "--" 없음
```

**SVG 파싱 에러를 유발한다.**

### SVG-05 — 골격 divider 좌표 (ERROR)

```
조건:  §3.8 골격 사용 시
         divider1 (User/Kernel):  y == 250,  x 90 → 690
         divider2 (S/W ↔ H/W):    y == 350,  x 30 → 690
```

**판정**: y가 250±6 또는 350±6인 수평선 중 길이 400 이상인 것을 골격으로 본다.
골격을 안 쓰는 다이어그램에는 이 규칙을 적용하지 않는다.

출처: `diagram-patterns/skeleton-layer.md` §2

### SVG-06 — 소켓 접촉 (ERROR)

```
조건:  골격 divider 를 가로지르는 타원/원이 없을 것
       cy + ry ≤ divider_y     (아래변이 divider 에 접하거나 그 위)
```

**1px 오차도 잡는다.** `cy=135, ry=16, divider=150`이면 아래변이 151로 침범이다.

에러 메시지에 **고칠 좌표를 제시한다**: `cy=240 를 234 로 (cy = divider_y - ry)`

**적용 대상**: divider를 실제로 가로지르는 도형만 (`top < divider_y < bottom`).
멀리 있는 타원은 검사하지 않는다.

출처: `diagram-patterns/skeleton-layer.md` §4

### SVG-07 — 화살촉 공식 (WARN)

```
조건:  3점 polygon 이 §3.2 공식을 만족
         팁 ~ 밑변중점 거리 == 12
         밑변 반폭 == 5
       (허용 오차 1.5px)
```

**예외 — 통과시킨다:**
- 수평·수직 표준 화살촉 (12×11, 10×5 규격)
- 3점이 아닌 polygon (화살촉이 아니다)

**WARN인 이유**: 검증되지 않은 변형이 있을 수 있다. ERROR로 막으면 false positive
위험이 크다.

출처: `diagram-patterns/arrowhead.md` §3

### SVG-08 — 팔레트 (INFO)

```
조건:  없음. 기록만 한다.
```

Core 밖의 색을 **누적 카운트**한다.

```
make palette-report

미등록 색:
  #eef7e0   4회   ← 3회 이상. 승격 검토
  #b19cd9   1회   ← 단발
```

**3회 이상 등장하면 승격을 검토한다.** 자동 등록하지 않는다 —
**"이 색이 무엇을 의미하는가"를 정의해야 Core에 들어간다.**

**WARN이 아니라 INFO인 이유**: 단발 색은 정상이다. 경고를 뱉으면 노이즈가 되고,
노이즈는 린터를 죽인다.

출처: `diagram-system.md` §2.2

### SVG-09 — 파일명 (WARN)

```
조건:  소문자 · 언더스코어 · 숫자만
경로:  diagrams/<domain>/<name>.svg
```

**prefix는 강제하지 않는다.** 디렉토리가 도메인을 말한다.

## 발행글 — `lint_post`

출처: `categories.md`, `tags.md`, `writing.md`

**카테고리·태그는 초안 frontmatter 에서 읽는다** (`--category`/`--tags` 로 덮어쓸 수 있다).
`make check` 가 초안을 린트하므로 POST-01~05 가 인자 없이도 강제된다. frontmatter 도
인자도 없으면 **조용히 넘기지 않고** "미검사" 를 알린다. `make build` 는 발행본에서
frontmatter 를 떼어낸다 (Tistory 오염 방지).

### POST-03 — Kind 태그 (ERROR)

### POST-04 — 카테고리 동명 태그 (ERROR)

```

```

카테고리는 도메인, 태그는 횡단 조회. **역할이 겹치면 안 된다.**

출처: `tags.md` §6

### POST-05a — 복수형 (WARN)

```
조건:  태그가 복수형이 아니다        Arrays ✗ → Array
```

**휴리스틱이다. 오탐이 난다.** 그래서 ERROR가 아니라 WARN이다.

`s`로 끝나지만 복수형이 아닌 태그가 많다:

```
Redis · Kubernetes · HTTPS · macOS · DNS · Windows · AWS · iOS
Emacs · Analytics · News · SaaS · TLS · DDoS · CORS · CSS
```

**판정:**

```
1. NOT_PLURAL 허용 목록에 있으면          → 통과 (조용히)
2. 's' 로 끝나지 않으면                   → 통과
3. 'ss' 'us' 'is' 'os' 로 끝나면          → 통과 (Class · Bus · Axis · macOS)
4. 나머지 's' 끝                          → WARN
```

**WARN 메시지에 등록 경로를 넣는다:**

```
[WARN] POST-05a: 'Redis' 가 복수형으로 보인다. 'Redi' 를 의도했나?
       고유명이면 NOT_PLURAL 에 등록하라.
```

#### promote — WARN을 confirm하면 등록한다

`NOT_PLURAL`은 **자라는 목록**이다. 처음부터 완전할 수 없다.

WARN이 뜨면 세 갈래로 판정한다.

| 결정 | 언제 |
|---|---|
| **fix** | 실제 복수형이다. `Arrays` → `Array` |
| **promote** | 고유명이다. **`NOT_PLURAL`에 등록한다.** 다음부터 조용해진다 |

**팔레트 승격 모델(SVG-08)과 같은 구조다.** 미등록 색이 반복되면 Core로 승격하듯,
고유명이 WARN을 받으면 허용 목록으로 승격한다.

> **WARN이 반복되면 규칙이 잘못된 것이지 사람이 잘못한 게 아니다.**

### POST-05b — 주관적 태그 (ERROR)

```
조건:  태그 ∉ { 추천, 꿀팁, 핵심, 필수, 완벽, 총정리 }
```

**유한 목록이다. 오탐이 없다.** 그래서 ERROR다.

검색어가 아니다. 태그는 독자가 검색창에 칠 말로 짓는다.

### POST-05c — 개인 분류 태그 (ERROR)

```
조건:  태그 ∉ { 기억할것, 공부, 나중에 }
       태그가 'type/' 등 경로 형식이 아님
```

**유한 목록 + 형식 검사다. 오탐이 없다.** 그래서 ERROR다.

Obsidian용 분류 태그이지 블로그용이 아니다.

출처: `tags.md` §6

### POST-06 — 죽은 링크 (ERROR)

**lychee를 쓴다. 자작하지 않는다.**

```bash
lychee posts/
```

lychee가 없으면 안내를 출력하고 **이 검사만 스킵한다.**
**네트워크가 정답성의 의존이 되면 안 된다** — 오프라인에서도 나머지는 돌아야 한다.

### POST-07 — 이미지 참조 (ERROR)

조건:  ![alt](name.png) 의 name.png 가 diagrams/ 아래에 존재
       alt 텍스트가 비어 있지 않음  (비면 WARN)
```

### POST-08 — 섹션 깊이 (WARN)

```
조건:  #    제목에만
       ##   최상위 섹션
       ###  하위 절
       #### 이하 없음
```

> **실측** (발행글 6편): **6/6 준수.** 이건 내가 만든 규칙이 아니라
> 이미 지켜지고 있던 관행을 발견한 것이다.

출처: `writing.md` §3.1

### POST-09 — 마무리 (WARN)

```
조건:  "## 마무리" 또는 "## 정리" 로 끝난다
```

> **실측** (발행글 6편): 5/6 준수. 예외는 `## 설치 완료`로 끝나는 글인데,
> **그 섹션이 마무리 역할을 하고 있다.**
>
> **ERROR였다면 멀쩡한 글이 차단됐다.** WARN이 맞다.

**요약이 아니라 재정리다.** 본문을 반복하지 말고 **읽는 법**을 한 줄로 압축한다.

출처: `writing.md` §3.3

### POST-10 — 본문 대시 (WARN)

```
조건:  본문 산문에 "—" 없음
```

**허용 — 검사 대상이 아니다:**

| 위치 | 예 |
|---|---|
| 제목·섹션 부제 | `# NAT — 하나의 주소로 여럿이 쓰는 법` |
| IMG 라벨 | `[IMG: Symmetric NAT #1 — 출발지 IP...]` |
| 표 셀 | `\| 값 — 설명 \|` |

**금지 — 본문 산문:**

```
✗ Inline, Out of path, Proxy—이 셋 중 하나에 속한다
✓ Inline, Out of path, Proxy. 이 셋 중 하나에 속한다
```

> **실측**: 발행글 6편에서 대시 77회 중 **본문 산문은 11회뿐이다.**
> 나머지 66회는 구조적 구분자다. **"대시 금지"로 뭉뚱그리면 false positive가
> 66건 난다.**

출처: `writing.md` §4.1

### POST-11 — 코드 블록 언어 태그 (WARN)

```
조건:  ``` 뒤에 언어 명시
```

### POST-12 — 극적 수사 어휘 (WARN)

```
조건:  본문 산문에 DRAMATIC_IDIOMS 목록의 어휘 없음
경로:  writing.md §4.3 <!-- DRAMATIC_IDIOMS:BEGIN/END -->
```

"영화 대본·드라마 대사" 톤을 막는다. **자라는 목록을 문서에서 파싱한다** (POST-01·05a
와 같은 계약 — 코드에 상수로 박지 않는다). 어휘 매칭이라 오탐이 나므로 WARN.
반복되면 목록에 promote 한다.

넓은 "톤"은 여전히 사람이 본다 ("검사하지 않는 것" 참고) — 여기 있는 **특정 관용구만**
기계가 잡는다. POST-10(본문 대시)·POST-05b(주관적 태그)와 같은 계열이다.

출처: `writing.md` §4.3

### POST-13 — `[IMG:]` 자작 다이어그램 파일 참조 (WARN)

```
조건:  [IMG: <name> ...] 의 <name>(소문자_밑줄 복합어)이
       diagrams/ 아래 <name>.{svg,gif,png} 로 존재
```

**POST-07은 발행본의 `![]()` 만 본다.** 초안이 실제로 쓰는 `[IMG:]` placeholder 의 자작
다이어그램 파일명은 그동안 미검사였다 — 파일명 드리프트(예: `dsa_dp_fib_tree` vs 실제
`dp_fib_tree`)가 발행본까지 새어나갔다.

**첫 토큰이 파일명 꼴(소문자_밑줄 복합어)일 때만 검사한다.** 산문 설명형(`[IMG: 설치 화면]`·
`[IMG: Symmetric NAT #1]`)은 첫 토큰이 파일명 꼴이 아니라 **건너뛴다.** 휴리스틱이라 WARN.

출처: `writing.md` §3.5

## Callout — `convert_callouts`

출처: `callouts.md`

변환기이지 린터가 아니다. **아래는 변환 계약이다.**

### CONV-01 — 코드 펜스 보호

```
조건:  ``` 안의 callout 문법을 변환하지 않는다
```

**가이드 문서 자체가 callout 예시를 코드 블록에 담고 있다.**
변환하면 문서가 망가진다. **놓치면 조용히 데이터가 깨진다.**

### CONV-02 — 인라인 마크다운 → HTML

```
`코드`      → <code>
**굵게**    → <strong>
*기울임*    → <em>
[텍스트](url) → <a href="...">
```

**Tistory는 `<blockquote>` 안의 마크다운을 파싱하지 않는다** (실측 2026-07-11).
백틱을 그대로 두면 리터럴로 노출된다. **변환기가 미리 `<code>`로 바꿔놔야 한다.**

### CONV-03 — LaTeX 보호

```
조건:  $...$ 안을 건드리지 않는다
```

**변환 순서:**
```
1. $...$ 를 placeholder 로 치환
2. HTML 이스케이프
3. 인라인 마크다운 변환
4. placeholder 복원
```

**검증된 버그**: 보호하지 않으면 `$a * b * c$` → `$a <em> b </em> c$`.
수식 안의 `*`가 기울임으로 오인된다.

### CONV-04 — alias 정규화

27개 alias → 8종 클래스. 표는 `callouts.md` §4.

### CONV-05 — 기본 타이틀

타이틀 생략 시 한글 기본값 부여. 표는 `callouts.md` §3.

### CONV-06 — 중첩 callout

```
조건:  outer 만 변환. inner 는 평문화 + 경고
       [!tip] 마커를 본문에 노출하지 않는다
```

## 팩트체크 — `factcheck`

출처: `writing.md` §2

린터가 아니라 **크로스 프로바이더 판정**이다. GPT가 판정하고 Claude가 조율한다.

판정은 두 경로 중 하나로 GPT 에 넘긴다. 어느 쪽이든 **판정자는 GPT(다른 프로바이더)** 라 FACT-02 를 지킨다.
- **수동**: `make factcheck` → 프롬프트를 ChatGPT 에 붙여넣고 응답을 `response.json` 에 저장.
- **자동**: `make factcheck-gpt` → `codex`(OpenAI GPT)를 read-only 로 실행해 `response.json` 을 바로 쓴다.
  codex 가 없으면 안내하고 멈춘다 (수동 경로로 폴백). **Claude 가 판정하지 않는다** — 그러면 에코 챔버다.

### FACT-01 — 판정 카테고리

```
VERIFIED      출처가 명시돼 있고 그 출처가 주장을 지지함
CONTRADICTED  출처와 본문이 어긋남
UNSOURCED     사실처럼 서술됐지만 출처가 없음
HEDGE_NEEDED  통념·경험담인데 단정형으로 쓰임
EXPERIENCE    1인칭 경험·환경 특정 관찰. 출처 불필요
```

**EXPERIENCE 가 결정적이다.** 기술 블로그에는 정당하게 출처 없는 문장이 많다.

```
"내 RTX 4060에서는 이 설정이 안 먹었다"
"내 PC의 IP가 192.168.0.10인데도 인터넷이 되는 이유는..."
```

**구분 못 하면 false positive 가 쏟아지고, 사람이 팩트체커를 무시하게 된다.**

### FACT-02 — 판정 기준

```
✗ "이 주장이 사실인가?"        ← GPT의 지식에 의존. 오염됨
✓ "글 안에 근거가 있는가?"      ← 텍스트 구조만 본다
```

**프롬프트에 반드시 넣는다:**

> 너 자신의 지식으로 "맞는 것 같다"고 판단하지 마라.
> 글 안에 근거가 제시돼 있는지만 본다.

**"출처 없이 지어냄"은 Claude와 GPT가 공유하는 실수다.**
그냥 물으면 GPT가 자기 훈련 데이터로 "맞네" 하고 넘긴다.
**프로바이더를 건너서도 에코 챔버가 발생한다.**

### FACT-03 — 라운드 캡

```
MAX_ROUNDS = 3
```

**루프는 언제나 여기서 종료된다.**

교착 시 **수렴한 척하지 않는다.** 미해결 주장을 명시하고 사람에게 넘긴다.

> **플래그된 불일치가 가짜 승인보다 낫다.**

### FACT-04 — 자동 수정 금지

**판정자는 조언한다. 명령하지 않는다.**

- GPT의 지적을 거부할 수 있다. 단 **이유를 로그에 남긴다**
- 자동 수정도 자동 통과도 없다
- 검증 불가능한 주장은 **삭제가 아니라 완화** ("~로 알려져 있다", "내 환경에서는")

## 검사하지 않는 것

**기계로 못 잡는다. 가이드에만 남긴다.**

| 규칙 | 왜 못 잡나 |
|---|---|
| "디바이스 드라이버로 통일" | 문맥 판단 필요 |
| "직각 꺾은선 선호" | 미학 판단 |
| "라벨은 빈 교집합에" | 렌더 결과를 봐야 안다 |
| "다이어그램이 의미를 전달하는가" | **사람이 본다** (게이트 ②) |
| **강의 내 실습 주제 태그 금지** (`고추건조기`) | **문맥 판단 필요.** 그게 실습 주제인지 도메인 개념인지 기계는 모른다 |
| **썸네일 오브젝트 선정** | **Claude 가 판단한다.** 글을 읽어야 안다 — 룩업 테이블로 안 된다 (`thumbnails.md` §2.4) |
| 톤 (정의 직술 / 절차 안내 / 1인칭) | 의미 판단 |
| AI 티 (넓은 톤) | **`humanize-korean` 스킬이 잡는다** (`im-not-ai` repo). 단 특정 극적 관용구는 POST-12 가 잡는다 |
| 썸네일이 개념을 전달하는가 | **블라인드 테스트** |

**이 목록이 중요하다.** 린터가 만능인 척하면 사람이 검수를 안 하게 된다.
