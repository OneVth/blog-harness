# 썸네일

Tistory 썸네일 제작 규칙.

이 문서는 **명세**다. `thumbnail_prompt`가 여기 정의된 규칙을 프롬프트에 박아 넣고,
`thumbnail_check`가 산출물을 검증한다.

---

## 1. 목적

기술 개념 하나를 **직관적인 오브젝트 하나로 압축한 교육용 아이콘**을 만든다.

화려한 유튜브 썸네일도, 광고 배너도 아니다.

---

## 2. 최상위 원칙

### 2.1 한 개념 = 한 오브젝트

썸네일은 글의 핵심 개념을 **하나의 오브젝트**로 표현한다.

**오브젝트 선정은 판단이다. 룩업이 아니다.**

아래 §2.4는 **예시**이지 사전이 아니다. 대부분의 글은 표에 없다.

### 2.2 코드를 그리지 않는다

썸네일은 코드 스크린샷이 아니다.

| 금지 | 허용 |
|---|---|
| `int Add(int a, int b)` | 함수 블록 |
| `for(int i=0; ...)` | 메모리 블록 |
| `malloc(...)` | 주소 화살표, 분기 구조, Stack 구조 |
| `docker run -d ...` | 격리된 컨테이너 블록 |

### 2.3 설명보다 상징

글을 설명하는 **포스터**가 아니라, 핵심 개념을 상징하는 **아이콘**이다.

---

### 2.4 오브젝트 고르는 법

**Claude가 초안을 읽고 고른다.** GPT에게 맡기지 않는다 — 매번 다르게 해석한다.

#### 절차

```
1. 글의 핵심 개념 하나를 뽑는다
     제목이 아니라 개념이다.
     "Docker Engine 설치" (제목) → "컨테이너 격리" (개념)

2. 그 개념을 물리적 오브젝트 하나로 압축한다
     추상 개념을 눈에 보이는 것으로 바꾼다.
     "컨테이너 격리" → "격리된 블록들이 하나의 커널 위에 놓임"

3. 150px 로 줄여도 보이는지 자문한다
     디테일이 필요하면 잘못 고른 것이다.
```

#### 판정 질문

> **"이 오브젝트만 보고, 글 제목을 모르는 사람이 개념을 떠올릴 수 있는가?"**

떠올리지 못하면 다시 고른다. 이게 §8.1 블라인드 테스트의 기준이다.

#### 예시 — C · DSA

| 개념 | 오브젝트 |
|---|---|
| 배열 | 연속된 메모리 셀 |
| 반복문 | 순환 화살표 |
| 제어문 | YES / NO 분기 |
| 함수 | 호출 흐름 (입력 → 함수 → 결과) |
| 포인터 | 주소 화살표 |
| Stack Frame | 쌓여 있는 프레임 |
| 동적 할당 | 메모리 블록 이동 |
| 재귀 호출 | 위로 쌓이는 함수 프레임 |
| 메모리 구조 | 수직 메모리 맵 (Stack/Heap/Data/Code) |
| 문자열 | 연결된 문자 블록 |
| 해시 테이블 | 키가 슬롯으로 흩어지는 그림 |
| 트리 | 갈라지는 노드 |

#### 예시 — Infra · Tools · Network

**표에 없는 도메인이 대부분이다. 원칙으로 만든다.**

| 글 | 개념 | 오브젝트 |
|---|---|---|
| Docker Engine 설치 | 컨테이너 격리 | 격리된 블록들이 하나의 커널 위에 |
| Firecrawl 셀프호스팅 | 웹 → 데이터 변환 | 웹페이지가 파이프를 통과해 문서로 |
| tmux 도입 | 세션 안의 분할 | 하나의 창 안에 나뉜 패널들 |
| NAT | 주소 바꿔치기 | 안팎을 잇는 관문에서 라벨이 바뀜 |
| VPN | 공용망 위의 사설 터널 | 열린 공간을 관통하는 닫힌 통로 |
| 듀얼부팅 | 하나의 디스크, 두 OS | 갈라지는 부팅 경로 |
| 가상 메모리 | 주소 변환 | 가짜 주소가 진짜 주소로 매핑 |

#### 실패하는 선택

| ✗ | 왜 |
|---|---|
| Docker 고래 로고 | **브랜드 로고는 개념이 아니다.** 저작권 문제도 있다 |
| 터미널 스크린샷 | **코드를 그리지 않는다** (§2.2) |
| 서버랙 사진 | 너무 일반적이다. Infra 글 전부가 같아진다 |
| 개념 3개를 다 그림 | **하나로 압축하라** (§6.1) |

#### 선정 근거를 기록한다

프롬프트 파일 맨 위에 남긴다.

```
# 개념: 컨테이너 격리
# 오브젝트: 격리된 블록들이 하나의 커널 위에 놓임
# 근거: 글의 핵심이 "Desktop이 아니라 Engine인 이유" = VM 없이 커널 직결
```

**블라인드 테스트가 실패하면 이 기록을 보고 오브젝트를 다시 고른다.**
프롬프트를 고치는 게 아니라 **판단을 다시 한다.**

## 3. 캔버스

| 항목 | 값 |
|---|---|
| **비율** | **반드시 1:1** (정사각형) |
| 기본 해상도 | 1024 × 1024 |
| 고해상도 원본 | 2048 × 2048 |
| 금지 비율 | 16:9, 4:3, 3:2, 세로형, 와이드 배너 |

### 축소 대응 — 절대 조건

Tistory 목록에서 이미지는 **매우 작게** 표시된다.

제작 기준은 *"1024px에서 예쁜 이미지"*가 아니라

> **"150px로 줄여도 무엇인지 보이는 이미지"**

이다.

`thumbnail_check`가 150px 다운스케일 이미지를 자동 생성한다. **그걸 보고 판단한다.**
육안으로 상상하지 않는다.

## 5. 배경

**배경은 주인공이 아니다.** 배경 때문에 오브젝트가 묻히면 실패다.

- 밝은 단색 (아래 카테고리 배경색)
- 종이 질감 허용
- 장식 최소화

카테고리를 새로 만들면 이 표에 한 줄 추가한다.

**색 배정 기준**: **같은 섹션 안에서** 구분되면 된다. 사이드바에서 나란히 뜨는 것은
같은 섹션 글들이다. (전 섹션 내 ΔE ≥ 15 — CIE Lab 기준)

섹션이 다른 카테고리끼리 색이 가까운 경우가 있다 (Embedded ↔ OSS Tools).
**색이 구분의 유일한 수단이 아니기 때문에 감수한다** — 오브젝트가 다르면 구분된다.
색은 "이 계열이구나" 정도의 약한 신호다.

## 6. 오브젝트

### 6.1 개수

- 주 오브젝트 **1개**, 또는 2~3개
- **3개 초과 시 재설계**
- 화면의 **50~70%** 차지
- 화면 중앙 배치, 여백 충분히

### 6.2 색

**주색상은 1개.** 무지개 금지.

배경색과 조화를 이루되, 오브젝트가 배경보다 먼저 보여야 한다.

---

## 7. 텍스트

**최소한으로 쓴다. 텍스트가 오브젝트보다 먼저 보이면 실패다.**

| 허용 | 금지 |
|---|---|
| `main()`, `Add()` | "메모리와 포인터" |
| `0x1000`, `10` | "함수 응용" |
| `YES`, `NO` | "Call By Reference" |
| `malloc`, `free` | 블로그 제목 전체 |

식별자·값·분기 라벨만 허용한다. **문장은 금지다.**

---

## 8. 품질 체크리스트

제작 후 반드시 확인한다. **하나라도 NO면 재설계.**

| # | 확인 | 검증 |
|---|---|---|
| 1 | 150px로 축소해도 무엇인지 보이는가 | `thumbnail_check` (다운스케일 생성) |
| 2 | 텍스트를 제거해도 개념이 전달되는가 | 사람 |
| 3 | 오브젝트가 3개 이하인가 | 사람 |
| 4 | 배경보다 오브젝트가 먼저 보이는가 | 사람 |
| 5 | **제목을 모르는 사람이 봐도 개념이 느껴지는가** | **블라인드 테스트** |
| 6 | 게임 UI가 아니라 기술 개념 아이콘인가 | 사람 |

### 8.1 블라인드 테스트 (5항)

가이드 5항 — *"글 제목을 모르는 사람이 봐도 '아, 메모리네' 정도가 직관적으로
느껴지는가?"* — 를 기계화한 것이다.

**150px 이미지만 판정자에게 던진다. 제목·카테고리·생성 프롬프트를 주지 않는다.**

```
이 이미지가 무슨 기술 개념으로 보이나? 한 단어로 답하라.
그리고 이게 "게임 아이템 아이콘"처럼 보이나,
"기술 개념 아이콘"처럼 보이나?
```

답이 의도한 개념과 맞으면 PASS. "게임 아이템"이나 "모르겠다"면 REDESIGN.

**판정자에게 정답을 숨기는 것이 설계의 핵심이다.** 정답을 주면 사후 합리화가
일어난다 — *"Docker라니까 Docker처럼 보이네."*

이미지는 GPT가 만들고 **판정은 Claude가 한다.** 만든 쪽이 채점하면 에코 챔버가 된다.

## 9. 기계 검사 (`thumbnail_check`)

| 규칙 | 수준 | 검사 |
|---|---|---|
| 비율 | ERROR | 1:1이 아니면 실패 |
| 해상도 | ERROR | 1024 또는 2048이 아니면 실패 |
| 색상 수 | ERROR | dominant color가 과다하면 실패 (무지개 금지) |
| 배경 밝기 | WARN | 가장자리 샘플링. 어두우면 오브젝트가 묻힌다 |
| 150px 생성 | INFO | `<slug>.150.png` 산출 |

## 10. 파이프라인 — 누가 무엇을 정하나

```
make thumbnail-prompt POST=drafts/foo.md CATEGORY=Infra
      │
      ├─ thumbnail_prompt.py  →  프롬프트 뼈대 (결정론적)
      │                          스타일 · 색 · 규격 · 개수 · 텍스트 · 금지목록
      │                          오브젝트 자리는 비워둔다:  {{OBJECT}}
      │
      └─ Claude              →  초안을 읽고 {{OBJECT}} 를 채운다  (§2.4)
                                 선정 근거를 주석으로 남긴다
      ▼
thumbnails/foo.prompt.txt     완성된 프롬프트
      │
      ▼
   사람: GPT 에 복붙          ← 게이트 ③
      │
      ▼
thumbnails/foo.png
      │
      ├─ make thumbnail-check  →  기계 검사 (§9) + 150px 생성
      │
      └─ 블라인드 테스트        →  Claude 가 판정 (§8.1)
      ▼
   PASS / REDESIGN
```

### 역할 분리

| 주체 | 역할 |
|---|---|
| **하네스** | 규칙을 박는다. 스타일·색·규격은 협상 대상이 아니다 |
| **Claude** | **판단한다.** 오브젝트 선정, 블라인드 검증 |
| **GPT** | **그린다.** 판단하지 않는다 |
| **사람** | 게이트 ③ (복붙), 최종 사인오프 |

**GPT에게 오브젝트 선정을 맡기면 GPT가 판단자 겸 생성자가 된다.**
*"가이드가 있는데도 매번 다르게 해석한다"*가 정확히 그 문제였다.

**Claude가 판단하고 GPT가 그린다.** 그리고 **다른 세션의 Claude가 검증한다** —
프롬프트를 만든 컨텍스트가 없는 상태에서 이미지만 본다 (§8.1).
